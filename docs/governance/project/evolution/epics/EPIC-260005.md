- Epic ID: EPIC-260005
- Title: Adaptive duration input (desktop steppers + touch roller)
- Persona/Actor: someone editing a timer's minutes and seconds in the library view
- Value statement: For a person setting a timer duration, deliver input controls matched to their device — arrow steppers with a mouse, a native roller on touch — so adjusting minutes and seconds is fast and precise on both.
- Context: the timer library view. After the duration-validation fix, both fields are plain text inputs: the native spinner arrows are gone on desktop, and on touch the fields require the on-screen keyboard for two small numeric values.
- Problem statement: editing a duration is slower than it should be on both device classes — no quick increment/decrement with a mouse, and fiddly keyboard entry on a phone.
- Expected outcome: on fine-pointer devices each field gains `▲`/`▼` steppers plus ArrowUp/ArrowDown key support; on coarse-pointer devices the two fields become native `<select>` controls so the platform presents its own roller/picker. Existing validation (0–59 per field, total ≥ `00:01`) and the red / disabled-Select treatment are preserved.
- Success metric(s): a duration can be set without typing on both device classes; the existing validation and responsive behaviour show no regression.
- Non-goals: changing the duration bounds; building a bespoke wheel component; any change to the selected timer (stage) view.
- Constraints: `00:01`–`59:59` total and ≤ 59 per field (REQ-CR-260002); static hosting, no backend (REQ-CR-260001); must not regress the stage responsiveness work.
- Dependencies: the draft/validation model in `TimerCard.tsx` and `timer.ts`.
- Risks/assumptions: touchscreen laptops report a coarse pointer and will get selects — acceptable, since `<select>` is fully keyboard-operable; native select presentation differs between iOS (wheel) and Android (dialog list), which is accepted in exchange for native feel and free accessibility.
- Target window: next small release.
- Validation approach: Playwright checks under emulated fine **and** coarse pointers, plus re-running the existing duration-validation and stage-responsiveness suites for regression.

## Ideation Bullets

Resolved during the 2026-07-20 discussion:

- [x] **Detection signal** → `(pointer: coarse)` / `(hover: none)` via CSS and `matchMedia`, reactive to change. Not viewport width and not user-agent sniffing. Evidence: a desktop window narrowed to 360px still reports `pointer: fine`, so width-based detection would wrongly hand a mouse user the touch picker; UA sniffing misreports iPadOS as macOS.
- [x] **Desktop control** → custom `▲`/`▼` buttons on the existing text input, plus ArrowUp/ArrowDown keys. Not a revert to `type="number"`, which cannot distinguish an empty field from invalid text (the defect fixed immediately before this epic).
- [x] **Touch control** → two native `<select>` elements (0–59) replacing the text inputs on coarse pointers; the platform supplies the roller.
- [x] **Consequence for validation** → with selects, a per-field invalid value becomes impossible; the only reachable invalid state on touch is the `00:00` pair, which keeps the existing red + disabled-Select treatment.

Resolved at the plan-review gate (2026-07-20):

- [x] **Stepper bounds behaviour** → clamp at 0 and 59; no wrap-around and no carry into the adjacent field. A value already above the range steps from the clamped edge (`99` + `▼` → `58`).
- [x] **Reaching `00:00`** → allowed, and shown with the existing invalid state rather than blocking the control — consistent with the "don't block the user, show the problem" principle.
- [x] **Press-and-hold auto-repeat** → included: immediate step on press, then ~400ms delay before repeating at ~8 steps/second.
- [x] **Records to spawn** → `ADR-260003` for "adaptive input by pointer modality", and `REQ-FR-260002`'s acceptance criteria extended to cover both input modes (no new requirement).
