# Epic: EPIC-260003 Active countdown interaction

State is tracked in `epic-index.md`; task states live in the inline checklist below. This epic
is **frozen** (Done) — retained for history per `artifact-model.md`.

- Epic ID: EPIC-260003
- Title: Active countdown interaction
- Persona/Actor: someone running a single timer at a time
- Value statement: For a person running a countdown, deliver a focused selected-timer view with tap start/stop/resume and reset so they can run one timer at a larger size.
- Context: the second view, entered by selecting a timer from the library.
- Problem statement: a selected timer needs a large, touch-friendly running view distinct from the management list.
- Expected outcome: a selected timer view with back navigation, an enlarged square timer, tap start/stop/resume, a reset below it, finish-to-paused behavior, and clear visual states.
- Success metric(s): a user can start, stop, resume, and reset the selected timer by tapping, on desktop and mobile.
- Non-goals: add/edit/delete (library view, EPIC-260002); persistence and deployment (EPIC-260004).
- Constraints: only the selected timer may run; no edit/delete in this view; timer bounds per REQ-CR-260002.
- Dependencies: EPIC-260002 (a timer to select).
- Risks/assumptions: switching timers must preserve the previous timer's remaining time.
- Target window: second delivery slice.
- Validation approach: end-to-end select/tap/reset/switch flows on desktop and mobile; `npm run build`.

## Ideation Bullets

- [x] What controls appear in the selected timer view? → back, enlarged square timer, tap start/stop/resume, reset below; no add/edit/delete.
- [x] What happens at zero? → the timer returns to its initial paused state and can be started again with a tap.
- [x] What happens when switching timers? → the previous timer stops and keeps its remaining time.
- [x] How are states shown? → initial-paused, running, and stopped-paused are visually distinct; finished returns to initial-paused visual.
