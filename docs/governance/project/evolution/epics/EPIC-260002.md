# Epic: EPIC-260002 Multi-timer editing experience

State is tracked in `epic-index.md`; task states live in the inline checklist below. This epic
is **frozen** (Done) — retained for history per `artifact-model.md`.

- Epic ID: EPIC-260002
- Title: Multi-timer editing experience
- Persona/Actor: someone maintaining a set of named countdown timers
- Value statement: For a person who runs several timers, deliver a library view to add, name, edit, and delete timers so they can maintain a clear set of reusable timers.
- Context: the default screen of the app, where the user manages their timers before running one.
- Problem statement: without a management view the user cannot create, rename, retune, or remove timers.
- Expected outcome: a timer library view supporting add, edit (name + duration), delete, and selection, with product limits enforced.
- Success metric(s): a user can create multiple timers and edit/delete each independently within the limits.
- Non-goals: running/countdown interaction (EPIC-260003); persistence and deployment (EPIC-260004).
- Constraints: at most 10 timers, name ≤ 15 chars, duration `00:01`–`59:59` (REQ-CR-260002).
- Dependencies: EPIC-260001 (stack + skeleton).
- Risks/assumptions: edit and delete belong in the library view only, not the running view.
- Target window: first delivery slice.
- Validation approach: end-to-end add/edit/delete/select flows in the library view; `npm run build`.

## Ideation Bullets

- [x] Which actions live in the library view? → add, name, edit duration, delete, select (edit/delete are library-only).
- [x] How are limits enforced? → ≤ 10 timers, name ≤ 15 chars, duration within `00:01`–`59:59`.
- [x] Does selecting a timer start it? → no; selection only moves it to the active view.
