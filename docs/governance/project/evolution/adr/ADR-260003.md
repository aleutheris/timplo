# ADR-260003: Adapt duration input to pointer modality, not viewport width

Status: Accepted
Date: 2026-07-20
Owners: repository maintainer

## Context

The timer library view edits a duration as two numeric fields. Two device classes want
different controls: a mouse user wants quick increment/decrement arrows, and a touch user wants
a roller rather than the on-screen keyboard for two small numbers.

The app already branches layout on viewport width (`@media (max-width: 640px)`), so width was
the obvious signal to reuse. It is the wrong one: width does not imply touch. Measured in
Chromium, a desktop window narrowed to 360px still reports `pointer: fine` — width-based
branching would hand a mouse user the touch picker. User-agent sniffing is the other common
approach and is worse: iPadOS reports itself as macOS, and the heuristic needs perpetual
maintenance.

## Decision

Branch the duration control on **input modality** via the `(pointer: coarse)` media feature,
read reactively through `matchMedia` (`src/useCoarsePointer.ts`) so a device that gains or loses
a mouse updates live.

- Fine pointer → text input with `▲`/`▼` stepper buttons and ArrowUp/ArrowDown key support.
- Coarse pointer → two native `<select>` elements (0–59); the platform supplies its own roller
  (a wheel on iOS, a picker dialog on Android).

Layout-only concerns (sizing, spacing) continue to use width media queries. Modality decides
*which control*, width decides *how it is laid out*.

## Alternatives Considered

- Option A: Branch on viewport width — rejected; a narrowed desktop window would lose its
  steppers, as measured above.
- Option B: User-agent sniffing — rejected; misreports iPadOS as macOS and needs ongoing
  maintenance.
- Option C: Revert to `<input type="number">` for free native spinners — rejected; per the HTML
  value-sanitisation rules a number input reports `""` for any non-numeric text, so "empty" and
  "invalid" become indistinguishable. That is the exact defect fixed immediately before this
  epic, and the validation model depends on telling them apart.
- Option D: A bespoke scroll-snap wheel in a modal sheet — rejected for now; it gives full
  visual control but costs significantly more code plus focus-trapping and screen-reader work,
  for no functional gain over the native roller.

## Consequences

- Positive: each device class gets an idiomatic control; the native select brings platform
  behaviour and accessibility for free.
- Positive: a coarse-pointer field cannot hold an invalid per-field value at all — only 0–59 is
  selectable — so the sole reachable invalid state on touch is the `00:00` pair.
- Negative: the touch control's appearance is browser-controlled and differs between iOS and
  Android; we accept the inconsistency in exchange for native feel.
- Negative: two rendering paths for one field must both be tested and kept in sync.
- Risks: touchscreen laptops report a coarse pointer and will receive selects. Accepted, since
  `<select>` is fully keyboard-operable, so the user is never blocked — only offered a control
  tuned for a different input.

## Verification

Playwright coverage exercises both modalities: emulated fine pointer (stepper increments,
clamping at 0/59, no carry, arrow keys, press-and-hold repeat) and emulated coarse pointer
(60 options per field, selection commits, `00:00` invalid handling, persistence). The
duration-validation and stage-responsiveness suites are re-run for regression.

## Rollback

Revert the commit. The decision touches presentation only — no stored data, no persisted schema,
and no change to the duration bounds — so reverting restores the plain text fields with the
validation model intact.
