# ADR-260002: Use a single active timer model with browser-local persistence

Status: Accepted
Date: 2026-05-13
Owners: repository maintainer

## Context

The product needs to support multiple named timers, immediate switching to one active countdown view, tap or click based pause and resume, reset behavior, and a static deployment model with no backend.

## Decision

Represent timers as a list with one selected timer at a time, keep selection separate from running state, allow only the selected timer to run, pause the previously selected timer when the user switches to another timer, drive start, pause, and resume from the active visual stage, and persist timer state in browser storage so the app restores across reloads on the same device.

## Alternatives Considered

Formal alternatives were not minuted at decision time (v1). Effectively set aside:

- Option A: Allow multiple timers to run concurrently — richer, but harder to reason about on a single-stage mobile view and heavier on ticking/battery.
- Option B: No persistence (in-memory only) — simpler, but loses timer state on refresh, which the responsiveness/persistence requirements reject.

## Consequences

- Positive: the UI can switch instantly between timers without server coordination; tap and click interactions behave consistently on desktop and mobile.
- Positive: selecting a timer only enlarges it into the active stage; it does not auto-start.
- Negative: only one countdown advances at a time, which is a deliberate product limitation, not a technical one.
- Risks: state restoration is local to one browser/device and is not shared or backed up.

## Verification

End-to-end checks cover switching selection while preserving remaining time, tap start/stop/resume on the active stage, reset behavior, and timer restoration after a page reload.

## Rollback

Revert to the prior timer model by reverting the commit; the localStorage key `timplo.timer-state.v1` can be bumped if a shape change makes old persisted state incompatible.
